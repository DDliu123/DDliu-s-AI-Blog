### 一级模块 2：Agent 开发前的核心准备工作

#### 二级要点 1：定义清晰的成功标准

- 核心要求：在开发前必须明确 4 个核心问题，避免目标模糊导致调优无方向：
    
    1. Agent 需要完成的核心任务是什么？
    2. 什么样的输出结果符合预期？
    3. Agent 需要哪些输入信息？
    4. Agent 需要输出什么标准化内容？
    
- 参考指引：推荐阅读 Claude 官方文档《定义你的成功标准》，这是 Agent 开发的前置核心前提。

#### 二级要点 2：搭建可量化的评测用例集

- 核心原则：**可测量是做 Agent 非常关键的一步**，只有可复现、可反复测试的 Agent，才能实现线上环境的稳定运行；量化评测可帮助开发者客观定位失败 case 的共性问题。
- 评测体系搭建的标准 5 步循环流程：
    
    1. 明确 Agent 的核心目标、效果要求、输入输出规范（以归因为例：目标是基于测试报告定位失败步骤并说明原因，效果要求是定位准确、原因清晰简洁，输入为测试报告、操作快照、测试步骤）
    2. 准备标准化评测集：包含对应场景的真实输入数据，以及人工标注的预期输出结果
    3. 快速开发 Agent 第一版，只需实现基础输出结构即可，无需追求完美
    4. 全量评测集跑测，输出评估报告，搭建可视化统计表格
    5. 分析报告，定位失败 case 的原因与解决方案，循环迭代 3-5 步，直到达成预期指标
    
- 关键提醒：必须覆盖边缘场景（edge cases），避免本地测试正常、线上环境效果翻车。

#### 二级要点 3：快速初始化第一版提示词

- 核心方法：**推荐使用 AI 快速生成第一版系统提示词**，而非人工手写，这也是 Claude 官方推荐的做法。
- 核心原因：
    
    1. 初始阶段对目标结果的认知是模糊的，人工手写效率极低，极易出现逻辑漏洞
    2. 借助 AI 可以快速完成初始版本，优先跑通「v0.1 Agent + 评测集 + 自动化跑测」的完整流程，而非纠结初版的完美度
    
- 实操示例：只需向 AI 明确 Agent 的核心目标、输入输出规范，即可快速生成初始系统提示词（如测试用例失败分析助手，明确 3 类输入数据与核心任务，AI 即可生成标准化的初始提示词）。

### 一级模块 3：提示词调优的核心方法

#### 二级要点 1：核心调优思路：用 AI 优化提示词

- 行业主流做法：借助 AI 完成提示词的迭代调优，尽量避免纯手动调整，核心原因有 2 点：
    
    1. 提示词调优是高频迭代的工作，从初版到阶段性终版往往需要几十轮调整，人工调整人力成本极高、效率极低
    2. 机器迭代的效率远高于纯手工调整，完美适配 Agent 快速迭代的开发节奏
    

#### 二级要点 2：通用调优提示词万能模板

- 模板核心逻辑：给 AI 明确 4 个必填模块，即可实现精准的提示词优化，避免调优方向偏差
- 模板核心 4 要素：
    
    1. 提供给 agent 的 input 输入内容
    2. Agent 当前返回的 output 输出内容
    3. 当前使用的完整 system-prompt 系统提示词
    4. 当前输出存在的问题与调优目标
    
- 格式规范：采用 XML 标签包裹各模块内容，确保 AI 能精准识别不同模块的边界，避免信息混淆。

### 一级模块 4：提示词调优实战 5 大核心经验

#### 二级要点 1：经验一：用好结构性提示词，提升 Agent 输出稳定性

##### 三级细节 1：明确核心目标（角色设定）

- 核心逻辑：通过目标 / 角色设定，限定 Agent 的工作边界，大幅提升输出表现
- 实战认知：并非所有 Agent 都需要强制角色设定，对于单一功能的 Agent，**清晰定义核心目标比强行赋予角色更重要**
- 正反示例：
    
    - 极简目标设定：“我会给你一个测试步骤执行的结果，以及他执行前后的快照，我希望你能帮我判断一下测试步骤执行是否符合预期。”
    - 角色 + 目标设定：“你是一个专业的测试用例失败分析助手，负责分析自动化测试执行失败的根因，并输出标准化、结构化的失败摘要与改进建议。你的结论将用于下游系统的程序化处理，因此必须严格遵循输出格式与字段含义。”
    

##### 三级细节 2：思维链 (CoT)/ 任务拆分

- 核心原理：将复杂任务拆分为更小的颗粒度，让 Agent 分步执行，大幅提升最终结果的准确性与可控性
- 实战效果：避免 Agent 直接输出片面结论（如直接把最后一个失败步骤当作根因），确保推理过程完整可追溯
- 实操示例：失败原因总结 Agent 的思维链设计，明确 4 步执行算法：
    
    1. 按顺序遍历 analysis 数组，不重排
    2. 对未通过的步骤，对比同序号历史成功步骤的表现，区分 “假失败” 与 “真失败”
    3. 锁定首个实际失败步骤，提取指定字段信息
    4. 按要求返回标准化 JSON 格式
    

##### 三级细节 3：使用 XML 格式组织提示词

- 核心优势 1：顺序无关性，无需严格保证提示词的先后顺序，通过标签即可精准引用内容
- 核心优势 2：灵活引用，可在提示词任意位置引用`<input>`等标签包裹的内容，无需固定位置
- 核心优势 3：局部更新便捷，可通过标签隔离固定内容（如示例、固定规则），避免修改提示词时影响无关内容

##### 三级细节 4：合理使用示例 (Few shot)

- 适用场景：当 Agent 在评测中已达到 60-70 分的基础准确率，通用提示词无法解决边缘 case 时，通过示例引导 Agent 行为
- 实战效果：针对性解决特殊场景的识别问题，提升极限准确率
- 实操示例：针对 URL 操作的特殊处理规则，通过输入输出示例，让 Agent 明确必须使用 aiAction 类型，而非 nativeUrl 类型

##### 三级细节 5：保持提示词精简

- 核心认知：系统提示词的上下文长度越大，Agent 的响应速度越慢，同时可能出现注意力分散的问题
- 取舍原则：思维链、示例等技巧不可无脑叠加，需在准确率与性能之间做平衡
- 实测数据：测试用例提速 Agent 精简提示词后，**平均响应时间从～20s / 次降至 < 10s / 次，tokens 消耗从～30k 降至～3k，效果保持不变**

#### 二级要点 2：经验二：严控系统提示词中的 Few shot，避免过拟合

- 核心问题：过多的 Few shot 会导致 Agent 过拟合，出现死板的固定化输出，泛化能力大幅下降，无法适配更多通用场景
- 实战踩坑案例：
    
    - 操作：在提示词中添加示例，说明 SG 开头的编号代表操作编号，希望 Agent 优化 css 选择器中的对应内容
    - 问题表现：Agent 出现过度泛化，无论 SG 编号出现在哪个字段（哪怕是输入内容的 value 值），都无脑替换为 “操作编号”，完全偏离预期
    
- 推荐做法：
    
    1. 初期零示例：Agent 开发初期，系统提示词尽量不要添加任何 Few shot，优先通过逻辑表述让 Agent 理解执行规则
    2. 严控添加时机：仅当系统提示词已能覆盖大部分 case，需要进一步提升极限准确率时，再添加极少量的 Few shot
    3. 模糊具体名词：编写示例时，尽量模糊掉商品名、特定编号等具体词汇，避免过拟合
    

#### 二级要点 3：经验三：精简输入，只给 Agent 最小必要信息

- 核心踩坑：初期给 Agent 输入完整的测试报告 + 全量测试步骤，希望一次性完成全流程优化，导致出现 4 类无解的核心问题：
    
    1. 分析不全：本该优化 5 个步骤，仅识别到 4 个
    2. 说谎 / 幻觉：声称已完成优化，实际未做任何修改
    3. 注意力不集中：优化步骤时引用了错误的上下文信息
    4. 捏造事实：无法完成优化时，凭空编造选择器等虚假信息
    
- 优化方案：
    
    1. 前置程序筛选：通过代码先做一轮预处理，仅将需要优化的步骤 + 对应的执行结果提供给 Agent
    2. 核心逻辑：让 Agent 无需做 “要不要优化” 的判断，只专注于 “怎么优化好” 这一件事
    
- 优化效果：
    
    1. 输出质量显著提升，上述问题基本解决
    2. 响应速度实现**10 倍提升**，从原先 > 60s 的响应时间，锐减至 6s
    

#### 二级要点 4：经验四：拆分更小的 Agent，遵循单一职责原则

- 核心原则：**一个 Agent 只做一件事，只承担一个最小职责**，这是目前行业主流的 Agent 开发实践
- 核心理由：如果一个 Agent 同时承担分析、生成、总结等多个职责，调优难度会指数级上升，效果与稳定性都无法保障
- 实战案例：测试用例失败归因场景，将原本的单一 Agent 拆分为 4 个职责单一的子 Agent，通过代码串联全流程：
    
    表格
    
    |Agent 名称|核心职责|
    |---|---|
    |分类师 (Classifier)|失败错误分类|
    |Diff 检查员 (Inspector)|Diff 步骤详细失败原因分析|
    |分析师 (Analyzer)|测试步骤执行结果分析|
    |协调者 (Coordinator)|基于 Diff 与执行结果，总结最终失败原因|
    
- 拆分后的核心收益：
    
    1. 单个 Agent 调优难度大幅降低，提示词可以更精准
    2. 整体响应速度更快
    3. 迭代影响面可控，修改单个 Agent 的提示词，不会影响全流程的其他环节
    

#### 二级要点 5：经验五：做好版本管理，建立合理的效果预期

##### 三级细节 1：做好 Agent 回滚准备，保管好老版本提示词

- 核心差异：Agent 开发与传统代码开发不同，**每一次系统提示词更新都是对上一版本的全部推翻**，极易出现负优化
- 实操要求：必须创建专门的文件，留存每一个稳定版本的提示词，出现负优化时可快速回撤

##### 三级细节 2：接受无法做到 100 分的事实，甚至 80 分

- 实战规律：Agent 的绝对采纳率达到 70% 之后（甚至 60%+），后续的提示词调整大概率会出现负优化，极易卡在瓶颈位，甚至出现采纳率大幅下跌
- 核心认知：Agent 开发比 E2E 测试难度更高，E2E 测试可通过控制变量实现 100% 通过率，但 Agent 调优中，追求 100% 准确率必然会导致过拟合
- 优化建议：当提示词调优达到瓶颈后，不要再局限于提示词本身，可通过 ReAct、RAG、N 次最佳验证等方法进一步提升效果
- 补充方法：N 次最佳验证，即多次使用相同提示词运行模型并对比输出，识别不一致的内容，规避幻觉问题
