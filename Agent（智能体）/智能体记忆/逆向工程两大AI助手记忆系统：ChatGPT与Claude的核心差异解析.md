# 一、ChatGPT的记忆系统：简洁高效的四层分层架构

经过大量实验，我发现ChatGPT的记忆系统远比预想中简单。它既未采用向量数据库，也未使用基于对话历史的检索增强生成（RAG）技术，取而代之的是四个功能明确的层级：适配使用环境的会话元数据、长期存储的显式事实、近期对话的轻量级摘要，以及当前对话的滑动窗口。这种分层设计在个性化、性能与令牌效率之间实现了精妙平衡。

## 1.1 ChatGPT的上下文结构

要理解记忆系统，首先需要明确ChatGPT处理每条消息时接收到的完整上下文。其结构如下：

```Plain
[0] 系统指令
[1] 开发者指令
[2] 会话元数据（临时）
[3] 用户记忆（长期事实）
[4] 近期对话摘要（过往聊天，含标题+片段）
[5] 当前会话消息（本次聊天）
[6] 你的最新消息
```

前两个部分定义了高层级的行为规范和安全规则，并非本文重点。真正值得关注的内容从会话元数据开始。

## 1.2 会话元数据：实时适配环境的临时信号

这些细节会在会话开始时注入一次，不会永久存储，也不会成为长期记忆的一部分。该模块包含以下内容：

- 设备类型（桌面端/移动端）
    
- 浏览器及用户代理
    
- 大致位置/时区
    
- 订阅级别
    
- 使用模式及活动频率
    
- 近期模型使用分布
    
- 屏幕尺寸、深色模式状态、JavaScript启用情况等
    

会话元数据示例如下：

```Plain
会话元数据：
- 用户订阅：ChatGPT Go
- 设备：桌面端浏览器
- 浏览器用户代理：macOS（Intel）上的Chrome
- 大致位置：印度（可能使用VPN）
- 当地时间：约16:00
- 账号年龄：约157周
- 近期活动：
    - 过去1天内活跃1天
    - 过去7天内活跃5天
    - 过去30天内活跃18天
- 对话模式：
    - 平均对话深度：约14.8条消息
    - 平均用户消息长度：约4057个字符
    - 模型使用分布：
        * 5% gpt-5.1
        * 49% gpt-5
        * 17% gpt-4o
        * 6% gpt-5-a-t-mini
        * 等
- 设备环境：
    - JavaScript已启用
    - 深色模式已启用
    - 屏幕尺寸：900×1440
    - 页面视口：812×1440
    - 设备像素比：2.0
- 当前会话时长：约1100秒
```

它们的作用，是当场用来“**判断你此刻的状态**”。换句话说，ChatGPT 会根据你当下的设备和使用习惯，来动态调整怎么说话。比如你是半夜一点，用手机打开它，它就可能默认你不想听废话，直接说重点。又或者，你最近常来，问题问得也复杂，它就知道你不是路人，是个“老用户”，说话自然会更配合一点。这些信号不构成“记忆”，但它极大影响了你的体验。

## 1.3 用户记忆：长期积累的个性化档案

ChatGPT拥有专门的工具用于存储和删除关于用户的稳定长期事实。这些信息会在数周、数月内不断积累，形成一个持久的“用户档案”。

以我为例，模型存储了33个事实，包括：

- 我的姓名、年龄
    
- 职业目标
    
- 背景及过往职位
    
- 当前项目
    
- 正在学习的领域
    
- 健身计划
    
- 个人偏好
    
- 长期兴趣
    

这些事实并非模型猜测得出，仅在以下两种情况下会被显式存储：

- 用户明确说“记住这一点”或“将这一点存入记忆”；
    
- 模型检测到符合OpenAI标准的事实（如你的姓名、职位或明确表述的偏好），且用户通过对话隐含同意存储。
    

这些记忆会作为独立模块注入未来的每一个提示词中。若你想添加或删除内容，只需直接说明：

> “将这一点存入记忆……” “从记忆中删除这一点……”

示例：

```Plain
- 用户姓名为Manthan Gupta。
- 曾任职于Merkle Science和Qoohoo（YC W23）。
- 偏好通过视频、论文和实操相结合的方式学习。
- 开发过TigerDB、CricLang、负载均衡器、FitMe。
- 正在研究现代信息检索系统（LDA、BM25、混合检索、稠密嵌入、FAISS、RRF、大语言模型重排序）。
```

## 1.4 近期对话摘要：轻量高效的跨会话连贯保障

这部分内容最让我意外，因为我原本以为ChatGPT会对过往对话使用某种检索增强生成（RAG）技术，没想到它采用的是轻量级摘要。ChatGPT以以下格式保存近期对话摘要列表：

```Plain
1. <时间戳>：<对话标题>
|||| 用户消息片段 ||||
|||| 用户消息片段 ||||
```

观察结果：

- 仅总结用户消息，不包含助手回复；
    
- 大约可获取15条摘要；
    
- 这些摘要仅作为近期兴趣的大致映射，而非详细上下文。
    

该模块让ChatGPT具备跨对话的连贯性感知，同时无需引入完整对话记录。传统检索增强生成（RAG）系统需要为每条过往消息生成嵌入向量、对每个查询执行相似度搜索、引入完整消息上下文，这会带来更高的延迟和令牌成本。而ChatGPT的方案更简洁：预计算轻量级摘要并直接注入。这种方式以牺牲详细上下文为代价，换取了速度和效率的提升。

## 1.5 当前会话消息：保障会话内连贯的滑动窗口

这是当前对话的常规滑动窗口，包含本次会话中所有交换消息的完整历史（未摘要）。我未能从ChatGPT中获取确切的令牌限制，但它确认了以下信息：

- 限制基于令牌数量，而非消息条数；
    
- 一旦达到限制，当前会话中较早的消息会被移除（但记忆事实和对话摘要仍保留）；
    
- 该模块中的所有内容均原封不动地传递给模型，以保证完整的对话上下文连贯性。
    

这也是助手能够在会话内进行连贯推理的关键。

## 1.6 ChatGPT记忆系统的整体工作流程

当你向ChatGPT发送消息时，具体流程如下：

1. 会话启动：会话元数据注入一次，为ChatGPT提供关于你的设备、订阅和使用模式的上下文；
    
2. 每条消息处理：你的存储记忆事实（我这边有33条）始终被包含在内，确保响应与你的偏好和背景相符；
    
3. 跨对话感知：近期对话摘要提供轻量级的兴趣映射，无需引入完整记录；
    
4. 当前上下文：当前会话消息的滑动窗口保证对话内的连贯性；
    
5. 令牌预算管理：随着会话推进，较早的消息会被移除，但记忆事实和对话摘要仍保留，确保连贯性不受影响。
    

这种分层方案让ChatGPT能够在不承担检索数千条过往消息的计算成本的前提下，实现个性化和上下文感知的体验。

# 二、Claude的记忆系统：按需检索的工具驱动架构

在剖析完ChatGPT的分层记忆架构后，我们将目光转向另一款主流AI助手Claude。通过同样的逆向工程方法，我发现Claude的记忆方案与ChatGPT截然不同——它并未采用预计算摘要注入的模式，而是构建了按需工具调用与选择性检索的架构。同时，针对上一篇关于ChatGPT研究收到的“清晰说明研究过程”的反馈，本节将明确包含逆向工程的方法学及所使用的具体提示词。

## 2.1 研究前提与注意事项

首先需要说明几个重要注意事项：

- Claude（与所有大语言模型一样）可能会产生幻觉，部分内容可能存在不准确之处。
    
- 相同的提示词对你而言可能无效，不同运行次数的响应结果也可能存在差异。
    
- 值得注意的是：Claude的记忆功能在免费版中不可用，因此本文所有实验均基于付费版完成。
    

## 2.2 研究方法：更顺畅的透明化探究

与逆向工程ChatGPT相比，一个显著差异是：Claude的配合度和透明度更高。由于Claude愿意分享其内部结构、工具及提示词格式相关信息，深入探究细节变得更加容易。这使得整个过程比ChatGPT顺畅得多——后者需要更持续的提示词引导才能获取信息。

本次研究采用的是简单且迭代式的方法：

6. 请求Claude描述其自身的提示词结构。

9. 寻找不同响应之间的一致性。

效果较好的方法：

- 直接询问工具签名
    
- 测试记忆的存储/删除行为
    
- 对比多个会话的响应结果
    

遇到的挑战：

- 不同会话间的部分响应存在差异
    
- 工具调用具有非确定性（由Claude决定何时使用工具）
    
- 确切的令牌限制和内部机制仍不透明
    

在可能的情况下，下文将附上所使用的具体提示词，以便你复现或调整研究过程。

## 2.3 Claude的上下文结构

## 2.4 用户记忆：类XML格式的长期事实存储

用户记忆相当于Claude的长期记忆，用于存储关于用户的精炼、稳定事实——这与ChatGPT的显式记忆事实类似，但存在一些关键差异。示例包括：

- 姓名和所在地
    
- 职业或角色
    
- 偏好和学习风格
    
- 使用的项目和工具
    
- 技术专业水平
    
- 长期对话中分享的个人细节
    

使用的提示词：

### 2.4.1 隐式更新

- 记忆会基于对话在后台定期更新
    
- 更新并非即时完成

### 2.4.2 显式用户编辑

## 2.5 对话历史：三大机制协同的选择性检索

Claude的对话历史功能比ChatGPT更复杂。它没有使用预计算摘要，而是通过三个协同工作的机制处理对话历史：

### 2.5.1 滚动窗口（当前对话）

- 包含完整消息（非摘要）
    
- 基于令牌限制（非消息数量限制）
    
- 达到限制时，较早的消息会被丢弃
    
- Claude称其总上下文预算约为19万个令牌
    

使用的提示词：

### 2.5.2 conversation_search工具

当Claude认为有必要时，可通过主题或关键词搜索过往对话。工具签名：

```Plain
{
  "query": "字符串（必填）",
  "max_results": "整数（1-10，默认：5）"
}
```

响应格式：


### 2.5.3 recent_chats工具

该工具基于时间检索近期对话。工具签名：

```Plain

  "n": "整数（1-20，默认：3）",
  "sort_order": "字符串（'asc'/'desc'，默认：'desc'）",
  "before": "时间戳（ISO格式，可选）",
  "after": "时间戳（ISO格式，可选）"
}
```

响应格式与conversation_search完全相同。

需要特别注意的是，conversation_search和recent_chats均不会在每次请求时自动调用。Claude似乎会选择性地使用这些工具——仅当它判定过往上下文相关时才会调用。这与ChatGPT的方案存在本质区别：ChatGPT会始终注入对话摘要。

潜在影响：

- 必要时，Claude可访问更详细的历史上下文
    
- 但这要求模型能够识别上下文的相关性
    
- 效率更高（不会在不相关历史上浪费令牌）
    
- 但可能存在连贯性不足的问题（若模型未检索到相关上下文，可能会遗漏关键信息）
    

使用的提示词：

> 你在对话历史中存储哪些内容？

> 你会从当前对话中注入多少条消息到提示词中？

> conversation_search和recent_chats工具的签名是什么？响应格式是什么？

## 2.6 Claude记忆系统的整体工作流程

当你向Claude发送消息时，具体流程如下：

11. 系统提示词：静态指令始终存在，用于定义行为和工具能力。
    
12. 用户记忆：你的存储事实以XML格式注入，提供关于你的持久上下文。
    
13. 当前对话：完整消息的滚动窗口确保会话内的连贯性。
    
14. 选择性检索：若Claude判定过往对话相关，会调用conversation_search或recent_chats工具获取特定上下文。
    
15. 你的消息：最终，结合所有上述上下文处理你的当前消息。
    

与ChatGPT不同——ChatGPT会预计算并注入对话摘要，Claude仅在需要时才检索历史上下文。这意味着Claude可在必要时访问更详细的信息，但要求模型能够准确判断检索时机。

# 三、ChatGPT与Claude记忆系统的核心差异与权衡

通过对两大AI助手记忆系统的逆向工程分析，我们可以清晰地看到两者截然不同的设计思路：ChatGPT采用预计算摘要注入的分层架构，而Claude则构建了工具驱动的选择性检索体系。两者的核心差异、对用户与开发者的影响，以及背后的工程权衡可总结如下：

## 3.1 核心差异对比

### 3.1.1 记忆管理

两者均使用显式长期事实存储用户个性化信息，但Claude具备更复杂的隐式更新和后台处理机制；ChatGPT的记忆更新则更依赖用户显式指令或模型对用户隐含同意的判定。

### 3.1.2 历史检索

ChatGPT会将轻量级的近期对话摘要注入每个提示词，确保所有对话的轻量化连贯性；Claude则通过conversation_search和recent_chats两个按需工具，仅在判定上下文相关时才检索历史——这种方式效率更高，但可能存在连贯性不足的问题。

### 3.1.3 设计理念

ChatGPT优先考虑预计算和注入，以牺牲部分历史深度为代价保障自动连贯性；Claude则优先考虑选择性检索和基于工具的访问，以牺牲部分自动连贯性为代价换取按需深度检索的能力。

## 3.2 对用户与开发者的影响

### 3.2.1 对用户的影响

Claude的方案意味着用户可在必要时获得更详细的历史上下文支持，但依赖模型对相关性的准确识别；ChatGPT的方案则能在所有对话中提供更顺滑的轻量化连贯性，但可能遗漏细微的历史细节。

### 3.2.2 对开发者的影响

Claude的基于工具的方案更灵活，潜在可扩展性更强，但对模型的检索时机判断能力提出了更高要求；ChatGPT的分层架构实现更简单，开发门槛更低，但灵活性相对有限。

## 3.3 工程权衡与适用场景

两者的权衡关系十分明确：Claude为了按需深度检索牺牲了自动连贯性；ChatGPT为了自动连贯性牺牲了历史深度。这两种都是合理的工程选择，“更优”方案完全取决于具体使用场景：

- 若需高频、轻量化的日常对话，追求稳定的连贯性体验，ChatGPT的方案更合适；
    
- 若需处理复杂任务、依赖详细历史上下文支撑深度推理，Claude的按需检索方案更具优势。
    

# 总结

ChatGPT与Claude的记忆系统，展现了主流AI助手在个性化体验与技术实现之间的两种典型路径。ChatGPT的四层分层架构，通过会话元数据、长期记忆、近期摘要和当前窗口的协同，实现了简洁高效的连贯性保障；Claude的工具驱动架构，则通过选择性检索解锁了深度历史上下文的按需访问能力。两者没有绝对的优劣之分，其设计差异本质上是对“连贯性”与“深度”的不同优先级选择，而这些选择最终服务于不同的用户需求与使用场景。

本次逆向工程研究也为我们提供了重要启示：优秀的AI记忆系统并非越复杂越好，而是要在个性化、性能、效率之间找到精准的平衡点。对于开发者而言，ChatGPT的务实分层设计与Claude的灵活工具化思路，均为AI记忆系统的构建提供了极具价值的参考范式。
