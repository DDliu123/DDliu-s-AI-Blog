# 产品经理如何写一个完整的需求文档

需求文档，也常被称为 PRD（Product Requirements Document），是产品经理把一个产品想法变成可执行方案的核心文档。它要讲清楚三件事：为什么做、给谁做、具体怎么做。

一份好的需求文档不是功能清单，而是团队协作的共同依据。它能帮助业务、产品、设计、研发、测试、运营等角色对齐目标、范围、优先级和验收标准。

## 1. 需求文档的核心作用

完整的需求文档通常有 5 个作用：

1. 对齐目标：让团队知道这个需求为什么重要。
2. 明确范围：说明本次做什么、不做什么。
3. 指导设计与研发：把用户流程、功能规则、页面状态写清楚。
4. 支持测试验收：让测试和产品知道如何判断需求是否完成。
5. 沉淀决策：记录背景、取舍、假设和版本规划，方便后续复盘。

## 2. 写需求文档前要先想清楚的问题

动笔前，产品经理应先回答以下问题：

| 问题 | 说明 |
| --- | --- |
| 解决什么问题 | 用户或业务当前遇到了什么困难？ |
| 为什么现在要做 | 是否有业务机会、用户反馈、数据异常、竞品压力或战略要求？ |
| 给谁使用 | 目标用户是谁？他们在什么场景下使用？ |
| 成功标准是什么 | 需求上线后，哪些指标证明它有效？ |
| 本次边界在哪里 | 哪些功能进入当前版本？哪些放到后续版本？ |
| 有哪些限制 | 时间、技术、合规、资源、数据、运营能力是否有限制？ |

如果这些问题没有想清楚，需求文档很容易变成“堆功能”，后续会导致返工、争议和延期。

## 3. 完整需求文档的推荐结构

一份完整的 PRD 可以按以下 8 个部分来写。

## 3.1 文档概述

这一部分用 2 到 3 句话说明文档目的。

建议包含：

- 需求名称
- 需求背景简述
- 本文档覆盖的功能范围
- 当前版本号和状态

示例：

```markdown
本文档描述「会员自动续费提醒」功能的产品需求。该功能用于在会员到期或续费前提醒用户，降低用户非预期扣费投诉，并提升续费流程透明度。本文档覆盖提醒规则、消息渠道、用户设置、异常处理和验收标准。
```

## 3.2 相关人员

这一部分用于明确负责人和参与方，避免后续沟通找不到人。

建议用表格表达：

| 角色 | 姓名 | 职责 |
| --- | --- | --- |
| 产品负责人 |  | 需求定义、优先级、验收 |
| 设计负责人 |  | 交互设计、视觉设计 |
| 研发负责人 |  | 技术方案、开发排期 |
| 测试负责人 |  | 测试用例、质量验收 |
| 运营负责人 |  | 上线配置、用户沟通 |
| 数据负责人 |  | 指标埋点、效果分析 |

## 3.3 背景与问题

背景部分要回答“为什么要做”。这部分不能只写一句“为了优化体验”，而要说明真实问题。

建议包含：

- 当前业务或用户场景
- 现有流程有什么问题
- 问题影响了哪些用户或指标
- 为什么现在需要解决
- 是否有用户反馈、数据证据或竞品参考

常见写法：

```markdown
当前用户在会员自动续费前缺少明确提醒，部分用户在扣费后才发现订阅仍在持续。这导致客服投诉增加，也影响用户对平台的信任。过去 30 天内，会员续费相关投诉占会员类投诉的 18%。因此需要在续费前建立清晰的提醒机制。
```

## 3.4 目标与成功指标

目标部分要说明“做到什么程度算成功”。目标要尽量具体，避免只写“提升体验”“提高转化”。

可以使用 SMART 原则：

- Specific：具体
- Measurable：可衡量
- Achievable：可实现
- Relevant：和业务相关
- Time-bound：有时间范围

建议包含：

- 产品目标
- 用户价值
- 业务价值
- 核心指标
- 反向指标

示例：

| 类型 | 指标 |
| --- | --- |
| 核心指标 | 会员续费相关投诉率下降 20% |
| 用户指标 | 续费提醒消息点击率达到 15% |
| 业务指标 | 自动续费取消率不高于当前基线 5% |
| 反向指标 | 提醒消息退订率不超过 3% |

反向指标很重要。它能防止团队只追求单一目标，却伤害用户体验或业务健康。

## 3.5 目标用户与使用场景

这一部分要回答“为谁做”。用户不能只按年龄、性别、城市等人口属性来定义，更应该按问题和任务来定义。

建议包含：

- 目标用户类型
- 用户当前任务
- 用户痛点
- 使用场景
- 触发条件

示例：

| 用户类型 | 场景 | 痛点 |
| --- | --- | --- |
| 已开通自动续费的会员用户 | 会员即将续费 | 不清楚扣费时间和金额 |
| 想管理订阅的用户 | 收到提醒后进入设置页 | 找不到取消或修改入口 |
| 客服人员 | 用户咨询扣费问题 | 缺少清晰的系统记录 |

## 3.6 价值主张

价值主张要说明这个需求给用户和公司分别带来什么价值。

可以从 3 个角度写：

1. 用户要完成什么任务？
2. 用户能获得什么收益？
3. 用户能避免什么损失或麻烦？

示例：

| 对象 | 价值 |
| --- | --- |
| 用户 | 提前知道续费时间、金额和管理入口，减少意外扣费感 |
| 公司 | 降低投诉和退款压力，提升用户信任 |
| 客服团队 | 更容易查询提醒记录和扣费状态，减少沟通成本 |

## 3.7 解决方案

解决方案是 PRD 的主体。它要把功能讲到研发、设计、测试都能执行。

通常包括以下内容。

### 3.7.1 用户流程

用户流程说明用户从哪里进入、做什么操作、系统如何响应。

可以用文字、流程图或表格表达。

示例：

```markdown
1. 系统在会员自动续费前 3 天判断用户是否符合提醒条件。
2. 若用户符合条件，系统发送站内信和短信提醒。
3. 用户点击提醒消息，进入会员管理页。
4. 用户可查看续费金额、扣费时间、支付方式。
5. 用户可选择继续保留自动续费，或进入取消自动续费流程。
```

### 3.7.2 功能列表

功能列表要说明每个功能点的优先级。

| 模块 | 功能 | 优先级 | 说明 |
| --- | --- | --- | --- |
| 提醒规则 | 续费前提醒 | P0 | 续费前 3 天发送提醒 |
| 消息渠道 | 站内信提醒 | P0 | 所有自动续费用户默认发送 |
| 消息渠道 | 短信提醒 | P1 | 仅对绑定手机号用户发送 |
| 用户设置 | 提醒偏好设置 | P2 | 后续版本支持 |

优先级可按以下方式定义：

- P0：没有它，需求不能上线。
- P1：重要，但可以在资源紧张时延后。
- P2：锦上添花，适合后续迭代。

### 3.7.3 页面与交互说明

页面说明要覆盖页面结构、字段、按钮、状态和跳转。

建议每个页面按以下格式写：

```markdown
### 页面名称：会员管理页

页面目标：
让用户清楚了解当前会员状态、续费时间、续费金额和管理入口。

入口：
- 会员中心
- 续费提醒消息
- 订单详情页

页面内容：
- 当前会员类型
- 到期时间
- 自动续费状态
- 下次扣费时间
- 下次扣费金额
- 支付方式
- 取消自动续费按钮

交互规则：
- 点击「取消自动续费」后，进入二次确认页。
- 若支付渠道不支持站内取消，展示对应渠道的操作说明。
- 若用户已经取消自动续费，不展示取消按钮。
```

### 3.7.4 业务规则

业务规则是最容易遗漏、也最容易引发返工的部分。

建议覆盖：

- 触发规则
- 判断条件
- 权限规则
- 时间规则
- 金额规则
- 状态流转
- 异常处理
- 边界情况

示例：

| 规则类型 | 规则说明 |
| --- | --- |
| 触发规则 | 系统每天 10:00 扫描 3 天后将自动续费的用户 |
| 去重规则 | 同一续费周期内，同一用户只发送一次提醒 |
| 状态规则 | 已取消自动续费的用户不发送提醒 |
| 异常规则 | 若手机号为空，则只发送站内信 |
| 合规规则 | 消息内容必须包含扣费时间、金额和取消入口 |

### 3.7.5 数据与埋点

如果需求上线后需要评估效果，PRD 中必须提前写清楚数据口径。

建议包含：

- 需要埋哪些事件
- 每个事件何时触发
- 事件属性有哪些
- 核心指标如何计算
- 数据看板由谁负责

示例：

| 事件名称 | 触发时机 | 关键属性 |
| --- | --- | --- |
| renew_notice_sent | 续费提醒发送成功 | user_id、channel、renew_date |
| renew_notice_click | 用户点击提醒消息 | user_id、channel、message_id |
| auto_renew_cancel_click | 用户点击取消自动续费 | user_id、entry_page |
| auto_renew_cancel_success | 用户取消成功 | user_id、payment_channel |

### 3.7.6 权限、风控与合规

涉及账号、支付、隐私、内容、通知、交易等场景时，这部分必须写清楚。

建议检查：

- 是否涉及个人信息
- 是否涉及支付或扣费
- 是否需要用户授权
- 是否需要保留操作记录
- 是否有监管或平台规则要求
- 是否需要风控拦截

### 3.7.7 异常场景

异常场景决定了产品是否可靠。

常见异常包括：

- 网络失败
- 接口超时
- 用户状态变化
- 支付渠道异常
- 数据为空
- 重复提交
- 权限不足
- 老版本客户端不兼容

示例：

| 异常情况 | 系统处理 | 用户提示 |
| --- | --- | --- |
| 提醒发送失败 | 记录失败原因，进入重试队列 | 无需提示 |
| 用户未绑定手机号 | 不发送短信，只发送站内信 | 无需提示 |
| 支付渠道不支持取消 | 展示渠道操作说明 | 请前往支付渠道管理自动续费 |

### 3.7.8 非功能需求

非功能需求不是页面功能，但会影响系统质量。

常见类型：

- 性能：页面加载时间、接口响应时间
- 稳定性：失败重试、降级方案
- 安全：权限校验、数据加密
- 兼容性：客户端版本、浏览器、系统
- 可访问性：字体大小、颜色对比、读屏支持
- 可维护性：配置化、日志、监控

## 3.8 版本计划与上线方案

版本计划要说明“先做什么，后做什么”。

建议不要在 PRD 中写过细的绝对日期，除非项目已经排期确认。可以使用相对阶段。

示例：

| 阶段 | 范围 | 目标 |
| --- | --- | --- |
| V1 | 续费前站内信提醒、会员管理页展示 | 满足基础合规和用户告知 |
| V2 | 增加短信提醒、提醒记录查询 | 提升触达率和客服处理效率 |
| V3 | 支持用户自定义提醒偏好 | 提升个性化体验 |

上线方案建议包含：

- 灰度范围
- 回滚方案
- 运营配置
- 客服话术
- 数据观察周期
- 上线后复盘时间

## 4. 验收标准怎么写

验收标准要让测试、研发、产品对“完成”的理解一致。

推荐使用 Given-When-Then 格式：

```markdown
Given 用户已开通自动续费，且将在 3 天后续费
When 系统执行每日提醒任务
Then 用户应收到一条续费提醒站内信
And 站内信内容包含续费时间、金额和管理入口
```

也可以使用表格：

| 场景 | 前置条件 | 操作 | 预期结果 |
| --- | --- | --- | --- |
| 发送提醒 | 用户将在 3 天后自动续费 | 系统执行提醒任务 | 用户收到站内信 |
| 不发送提醒 | 用户已取消自动续费 | 系统执行提醒任务 | 用户不收到提醒 |
| 查看管理页 | 用户点击提醒消息 | 进入会员管理页 | 展示扣费时间和金额 |

## 5. 常见错误

产品经理写需求文档时，常见问题有：

- 只写功能，不写问题和目标。
- 只写正常流程，不写异常流程。
- 只写页面，不写业务规则。
- 只写“支持”“优化”“完善”，没有明确标准。
- 不写数据口径，导致上线后无法评估效果。
- 不写不做什么，导致范围不断扩大。
- 不写验收标准，导致测试和产品反复争论。
- 不标注假设，导致未经验证的判断被当成事实。

## 8. 结论

完整的需求文档应该从问题出发，而不是从功能出发。产品经理要先讲清楚“为什么做”和“给谁做”，再讲“怎么做”和“如何验收”。

判断一份 PRD 是否合格，可以看它是否让团队在没有反复口头解释的情况下，理解目标、范围、流程、规则、异常和验收标准。如果能做到这一点，它就不只是文档，而是推动产品落地的协作工具。
