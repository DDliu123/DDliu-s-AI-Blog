## 六个维度，逐层拆解

**① 问题与场景 — 先搞懂"为什么存在"**

从 README、Issues 第一页、Discord/Slack 社区入手。核心问题是：这个项目在哪个环节替代了什么？谁在用（个人开发者、企业、研究者）？用户反复提的 pain point 是什么？这是最容易被 PM 跳过但最重要的一层。

**② 核心能力 — 功能边界与技术架构**

看 `docs/`、`architecture.md`、核心模块的目录结构。重点不是看懂代码，而是搞清楚：项目的 MVP 是什么 → 后来加了什么 → 哪些功能在 roadmap 但还没做。功能演进顺序本身就是产品决策的证据。

**③ 交互设计 — 用户路径怎么设计的**

如果有 UI，直接 fork 跑起来体验。没有 UI 的项目，看 CLI 参数设计、API 接口命名、错误提示文案。这些细节反映了设计者对用户认知成本的判断。重点问自己：第一次用的用户需要几步才能跑通核心流程？

**④ 数据与模型 — AI 产品特有的一层**

看 model card、训练数据说明、推理配置。PM 关注的不是算法细节，而是：项目选了哪个模型/自己训练？为什么？在哪些场景做了 prompt engineering？这些决策直接影响产品的准确率上限和成本结构。

**⑤ 商业与生态 — 开源背后的逻辑**

看 License（MIT/Apache 还是 AGPL 直接影响商业化策略）、是否有 cloud 版本、sponsors 是谁、Star 增长曲线。很多开源项目是公司的获客漏斗，弄清楚"谁在养这个项目、目的是什么"，能帮你判断它的长期可靠性。

**⑥ 竞品对比 — 最能产生产品洞察的环节**

找 2-3 个解决同一问题的项目，列出它们在同一维度上的不同选择。比如同样是 AI 编程助手，A 选择了深度 IDE 集成，B 选择了纯 CLI，C 选择了 web UI — 这背后是对用户场景的不同判断。差异本身就是产品思路。

## 如何转化成日常工作资产

建议养成三个习惯：

**建立"设计决策库"** — 每次拆解后，把发现的有趣决策记成一条卡片，格式是"[项目] 在 [场景] 下选择了 [方案A] 而非 [方案B]，原因可能是 [推断]"。积累 50 条后，你会开始看到跨项目的模式。

**用 Issues 做需求研究** — GitHub Issues 是免费的用户访谈记录。搜索 `is:issue label:enhancement` 或 `is:issue "please add"` 找功能请求，看哪些需求被反复提、哪些被关闭及原因，这是最真实的需求验证场。

**跟踪版本更新** — Watch 项目的 Releases，每次更新看 changelog 里加了什么、删了什么、改了什么。迭代方向比当前功能更能说明团队的产品判断。
