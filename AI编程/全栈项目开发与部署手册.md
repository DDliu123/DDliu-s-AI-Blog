## 第一章：哪些个人项目适合这套方案？

在动手之前，先确认你的项目类型是否与这套技术栈匹配。以下是最适合的场景，以及少数不适合的情况。

### 1.1 最适合的项目类型

|项目类型|典型案例 / 说明|
|---|---|
|**SaaS 工具**|AI 写作助手、简历生成器、发票管理系统——有登录、订阅、后台数据存储|
|**内容平台**|博客、知识库、问答社区、Newsletter 平台——读多写少，CDN 加速效果显著|
|**电商 / 预约**|独立站卖家后台、餐厅在线预订、课程报名系统——需要支付集成和订单管理|
|**开发者工具**|API 测试平台、代码片段管理器、Webhook 调试器——技术用户，API 优先|
|**社区 / 论坛**|小型垂直论坛、兴趣圈子、粉丝站——用户生成内容，实时互动|
|**数据看板**|个人财务追踪、健身记录、习惯打卡——以可视化为核心，数据私有|
|**AI 应用**|基于 Claude / GPT 的对话机器人、文档问答、图片处理工具|

### 1.2 不太适合这套方案的项目

> [!warning] 以下场景建议选择专项方案
> 
> - **游戏（3D / 实时对战）**：需要 Unity / Unreal 或专用游戏服务器，本套方案过重
> - **纯静态官网**：直接用 Webflow / Framer，零代码更快
> - **强合规行业（医疗 HIS、金融核心系统）**：需要等保认证和私有化部署
> - **超大规模（DAU > 10 万起步）**：需要专职 DevOps 和更复杂的微服务拆分

### 1.3 项目规模与方案对应关系

根据预期用户量，选择不同的基础设施起点：

|阶段|DAU 参考|推荐方案|月成本估算|
|---|---|---|---|
|MVP 验证|0 – 500|Vercel + Supabase 免费层 + Railway|¥0 – ¥50|
|早期增长|500 – 5000|Vercel Pro + Supabase Pro + Fly.io|¥150 – ¥600|
|稳定运营|5000 – 50000|AWS / GCP + RDS + Redis + CDN|¥600 – ¥6000|
|规模扩张|50000+|Kubernetes + 多区域部署 + 专职 SRE|按需定制|

## 第二章：整体技术架构

本手册推荐的架构遵循「分层解耦」原则，六层各自独立，可按需替换其中任意一层。

### 2.1 架构全景

```
用户层（浏览器 / iOS / Android）
        ↓ HTTPS
CDN 层（Cloudflare / Vercel Edge）
        ↓
前端层（Next.js → Vercel）
        ↓ REST / GraphQL
API 网关（认证 · 限流 · 路由）
        ↓
后端层（Node.js / Python → Railway / Fly.io）
        ↓
数据层（PostgreSQL · Redis · 对象存储）
```

|层级|职责与推荐技术|
|---|---|
|① 用户层|浏览器、iOS / Android App——用户直接访问的端点|
|② CDN 层|Cloudflare / Vercel Edge——静态资源缓存、DDoS 防护、全球加速|
|③ 前端层|Next.js（React）部署到 Vercel——页面渲染、路由、用户界面|
|④ API 网关|认证鉴权（JWT / OAuth）、限流、路由转发——前后端的统一入口|
|⑤ 后端层|Node.js + Fastify 或 Python + FastAPI——核心业务逻辑|
|⑥ 数据层|PostgreSQL（Supabase）+ Redis（Upstash）+ 对象存储（R2）|

### 2.2 数据流说明

一次典型的用户请求完整路径如下：

1. 用户在浏览器输入 URL，DNS 解析到 Cloudflare
2. Cloudflare 检查缓存：命中则直接返回，未命中则转发到 Vercel
3. Vercel 边缘节点渲染或返回 Next.js 页面
4. 页面内 JavaScript 发起 API 请求（fetch / axios）到后端服务
5. API 网关验证 JWT Token，通过后路由到对应微服务
6. 后端查询 Supabase（PostgreSQL），高频数据先走 Redis 缓存
7. 后端返回 JSON 数据，前端渲染更新 UI

### 2.3 Claude Code 在架构中的角色

> [!tip] Claude Code 能直接帮你做什么
> 
> - **项目脚手架**：执行 `create-next-app`、初始化 Git 仓库、配置 ESLint / Prettier
> - **数据库 Schema**：根据需求描述生成 SQL 建表语句和 Supabase Migration 文件
> - **API 路由**：生成 RESTful 或 tRPC 接口、自动写单元测试
> - **Dockerfile**：针对你的后端框架生成多阶段构建的最优 Dockerfile
> - **CI/CD 配置**：生成 GitHub Actions `.yml`，实现推送即部署
> - **调试排错**：粘贴报错信息，Claude Code 直接定位并修改相关文件

## 第三章：推荐技术栈详解

### 3.1 前端：Next.js + Vercel

#### 为什么选 Next.js

- Next.js 由 Vercel 开发，原生支持 SSR、SSG、App Router，SEO 友好
- 内置 API Routes，小项目可以不单独部署后端
- 生态最完整：Tailwind CSS、shadcn/ui、Radix UI 可直接使用
- 与 Vercel 的配合是原生级别的：推送代码即自动构建部署

#### Vercel 免费套餐包含什么

- 无限个人项目，每月 100GB 流量
- 自动 HTTPS 证书（自动续签，无需手动操作）
- 每次 PR 自动生成预览链接（Preview Deployments）
- 全球 CDN 节点，国内访问需搭配自定义域名

#### 同类替代方案

- **Cloudflare Pages**：免费额度更大，边缘网络覆盖更广
- **Netlify**：功能与 Vercel 类似，表单处理更方便

### 3.2 后端：Node.js (Fastify) 或 Python (FastAPI)

#### Node.js + Fastify

- 与前端共享 TypeScript，减少上下文切换
- Fastify 比 Express 快约 2 倍，内置 Schema 验证（Zod / AJV）
- npm 生态庞大，AI / 支付 / 邮件等 SDK 应有尽有

#### Python + FastAPI

- 如果项目涉及机器学习、数据分析，Python 生态碾压性优势
- FastAPI 自动生成 OpenAPI 文档，接口调试极方便
- 异步支持好，适合 AI 流式输出（Streaming）场景

#### 部署到 Railway 或 Fly.io

- **Railway**：操作最简单，连接 GitHub 后一键部署 Docker 容器，按用量计费
- **Fly.io**：全球多区域部署，冷启动速度快，免费层有 3 个小型实例
- 两者都支持环境变量管理，避免密钥硬编码

### 3.3 数据库：Supabase (PostgreSQL)

#### Supabase 的核心优势

- 托管的 PostgreSQL，无需自己运维数据库服务器
- 内置认证系统（邮箱 / Google / GitHub 登录），直接用，无需自己写
- Row Level Security（行级权限），数据安全开箱即用
- 实时订阅（Realtime）：数据变化自动推送到前端，适合聊天、协作类功能
- 免费层：500MB 数据库存储，每月 5GB 流量

#### 数据库设计建议

- 每张表都加 `id`（UUID）、`created_at`、`updated_at` 字段
- 用 Supabase Migration 管理 Schema 变更，让 Claude Code 生成迁移文件
- 对高频查询字段建索引（如 `user_id`、`created_at`）

### 3.4 缓存层：Redis (Upstash)

- Upstash 是 Serverless Redis，按请求计费，免费层每天 1 万次请求
- 适合缓存：Session、API 限流计数器、热点数据、队列任务
- 早期项目可以跳过，DAU 超过 1000 后再引入

### 3.5 文件存储：Cloudflare R2

- 兼容 AWS S3 API，现有 S3 SDK 可直接用，无需改代码
- 免费层：10GB 存储 + 每月 100 万次读请求，适合早期项目
- 无出站流量费（这是 S3 最贵的部分），成本远低于 AWS
- 适合存储：用户上传的图片、PDF、视频、导出文件

## 第四章：CI/CD 与自动化部署

CI/CD 的目标是：你只需专注写代码，推送到 GitHub 后一切自动化完成。

### 4.1 GitHub Actions 基本流程

1. 开发者推送代码到 GitHub `main` 分支
2. GitHub Actions 触发 Workflow
3. 自动运行测试（单元测试、集成测试）
4. 测试通过后构建 Docker 镜像
5. 推送镜像到 GitHub Container Registry 或 Docker Hub
6. 触发 Railway / Fly.io 重新部署新版本
7. 部署完成后，Slack / 邮件发送部署通知（可选）

### 4.2 Claude Code 生成 CI/CD 配置

> [!example] 你只需要告诉 Claude Code： 「帮我写一个 GitHub Actions workflow，当推送到 main 分支时，自动运行 `npm test`，然后构建 Docker 镜像并部署到 Railway。Railway Token 存储在 GitHub Secrets 中。」
> 
> Claude Code 会直接在项目中创建 `.github/workflows/deploy.yml` 文件。

### 4.3 分支策略建议

- **`main` 分支**：生产环境，受保护，不允许直接推送
- **`develop` 分支**：开发集成分支，对应预发布环境
- **`feature/*` 分支**：每个功能一个分支，完成后 PR 合并到 `develop`
- Vercel 会自动为每个 PR 创建预览链接，方便 Review

## 第五章：域名、HTTPS 与安全

### 5.1 域名注册

- **国内推荐**：阿里云万网、腾讯云——`.com` 约 ¥60/年，续费需注意价格
- **国际推荐**：Cloudflare Registrar（成本价销售，无溢价）、Namecheap
- **选域名建议**：简短、好记、无连字符、优先 `.com` 或 `.io`

### 5.2 DNS 配置

- 将域名 NS 记录指向 Cloudflare（免费），统一在 Cloudflare 管理 DNS
- Vercel 前端：在 Vercel 控制台添加自定义域名，按提示添加 CNAME 记录
- 后端服务：添加 A 记录或 CNAME 指向 Railway / Fly.io 分配的地址

### 5.3 HTTPS 证书

- Vercel 和 Cloudflare 均自动签发 Let's Encrypt 证书，自动续签
- 开启 Cloudflare「Full (Strict)」模式，确保全链路加密
- 后端若部署在 Railway / Fly.io，这些平台也自动提供 HTTPS

### 5.4 基础安全清单

> [!danger] 上线前必须检查以下安全项
> 
> - [ ] 所有密钥（数据库密码、API Key）存在环境变量中，不提交到 Git
> - [ ] `.env` 文件已加入 `.gitignore`
> - [ ] API 接口有认证保护（JWT 或 API Key）
> - [ ] 数据库开启 Row Level Security（Supabase 默认支持）
> - [ ] 启用 CORS 白名单，只允许自己的前端域名调用 API
> - [ ] 依赖包定期更新（`npm audit` / `pip audit`）
> - [ ] 生产环境关闭调试信息和详细报错输出

## 第六章：监控、日志与运维

### 6.1 错误追踪：Sentry

- 免费层支持 5000 次错误/月，对早期项目绰绰有余
- 前端和后端都可以接入，一个平台看全所有报错
- 自动捕获未处理异常，包含完整的堆栈跟踪和用户上下文
- Claude Code 可以帮你生成 Sentry 初始化代码

### 6.2 性能监控

- **Vercel Analytics**：内置前端性能指标（Core Web Vitals），免费
- **Supabase Dashboard**：数据库查询性能、慢查询日志，内置
- **Uptime 监控**：使用 Better Uptime 或 UptimeRobot（免费）监控服务可用性

### 6.3 日志管理

- **开发阶段**：Railway / Fly.io 控制台直接查看实时日志
- **生产阶段**：接入 Logtail（Better Stack）或 Datadog，支持日志搜索和告警
- **结构化日志**：使用 JSON 格式输出，方便查询过滤

### 6.4 成本控制

> [!info] 避免意外超支的最佳实践
> 
> - 在云厂商控制台设置账单告警（Billing Alert）
> - **Supabase**：注意连接数限制，使用 PgBouncer 连接池
> - **Vercel**：Edge Function 调用次数按量计费，注意循环调用
> - **Cloudflare R2**：读请求免费，写请求按量计费——不要在循环中频繁写入
> - **Railway**：设置月度用量上限，防止流量突增导致账单暴涨

## 第七章：从零到上线的完整步骤

以下是一个典型 SaaS 项目从立项到上线的完整操作路径，可作为检查清单使用。

### 第一阶段：本地开发（Day 1 – 3）

- [ ] 注册 GitHub 账号，创建新仓库（Public 或 Private）
- [ ] 在终端安装 Claude Code：`npm install -g @anthropic-ai/claude-code`
- [ ] 进入项目目录，启动 Claude Code：`claude`
- [ ] 告诉 Claude Code：「帮我用 Next.js + Tailwind + Supabase 搭建一个 [你的项目] 的骨架」
- [ ] Claude Code 自动创建目录结构、安装依赖、配置 Supabase 客户端
- [ ] 在 Supabase 控制台创建项目，获取 API URL 和 anon key
- [ ] 配置 `.env.local` 文件，完成本地数据库连接
- [ ] 开发核心功能，随时用自然语言描述需求交给 Claude Code 实现

### 第二阶段：部署上线（Day 4 – 5）

- [ ] 将代码推送到 GitHub
- [ ] 登录 Vercel，连接 GitHub 仓库，一键部署前端
- [ ] 在 Vercel 控制台配置环境变量（Supabase URL、API Key 等）
- [ ] 创建后端服务的 Dockerfile（让 Claude Code 生成）
- [ ] 登录 Railway，连接 GitHub 仓库，部署后端服务
- [ ] 在 Railway 配置后端环境变量
- [ ] 在域名注册商购买域名，将 NS 指向 Cloudflare
- [ ] 在 Cloudflare 配置 DNS，绑定自定义域名到 Vercel 和 Railway

### 第三阶段：上线运维（Day 6+）

- [ ] 接入 Sentry，监控前后端报错
- [ ] 配置 GitHub Actions，实现推送自动部署

- [ ] 在云厂商控制台设置账单告警
- [ ] 根据用户反馈持续迭代，Claude Code 帮你快速实现新功能

## 第八章：进阶方案（用户量增长后）

早期不要过度设计。当遇到明确的性能瓶颈时，再针对性地升级对应层级。

### 8.1 数据库扩展

- **读写分离**：Supabase Pro 支持只读副本（Read Replica），分摊查询压力
- **引入 Redis 缓存**：热点数据缓存到 Upstash，减少数据库查询
- **分库分表**：DAU 超过 5 万后考虑，使用 Citus（PostgreSQL 分布式扩展）

### 8.2 后端扩展

- **水平扩展**：在 Fly.io 增加实例数量，配置自动扩缩容
- **微服务拆分**：将耗时任务（邮件发送、图片处理）拆分为独立 Worker 服务
- **消息队列**：引入 BullMQ（基于 Redis）处理异步任务

### 8.3 迁移到 AWS / GCP

- 后端迁移到 AWS ECS（容器）或 EKS（Kubernetes）
- 数据库迁移到 AWS RDS（PostgreSQL），开启多可用区部署
- 使用 AWS CloudFront + S3 替代 Cloudflare R2 处理媒体文件
- 这一步建议在有专职后端工程师后再做，不要提前优化

### 8.4 何时考虑自建服务器

> [!note] 自建服务器（VPS）适合这些场景
> 
> - 有大量计算密集型任务（AI 训练、视频转码）需要 GPU 服务器
> - 云服务成本超过 ¥5000/月，自建 VPS 性价比更高
> - 数据合规要求必须存储在境内私有服务器
> - 团队已有专职运维人员，能处理硬件和系统问题
> 
> 否则，继续使用托管云服务——节省的运维时间更有价值。
