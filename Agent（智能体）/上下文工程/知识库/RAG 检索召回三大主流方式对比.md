# 检索策略

在 RAG（检索增强生成）系统中，检索召回是连接 “外部知识库” 与 “大模型生成” 的核心环节，其效果直接决定了大模型获取信息的准确性和全面性。全文检索、向量检索、混合检索是当前主流的三种召回方式，三者在技术逻辑、适用场景、开源 / 商业方案上存在显著差异。

## 一、全文检索（Full-Text Retrieval）

全文检索是基于 “关键词匹配” 的传统检索方式，核心逻辑是通过分析文本中的 “字面特征”（如单词、短语），快速定位包含查询关键词的文档，不涉及文本语义的深层理解。其优势是速度快、可解释性强，适合对 “明确关键词” 的精准匹配；劣势是无法处理 “语义相似但关键词不同” 的场景（如 “如何缓解头痛” 与 “头痛怎么解决”）。

1. ### 技术实现：关键词匹配与索引优化
    

全文检索的核心是构建高效的文本索引，通过 “预处理 - 索引构建 - 查询匹配” 三步实现召回：

- Step 1：文本预处理 对知识库文档进行标准化处理，消除噪声并提取关键特征，常见操作包括：
    
    - 分词
        
        - 英文按空格分，如 “machine learning”→“machine”“learning”
            
        - 中文用 Jieba/IKAnalyzer，如 “人工智能应用”→“人工智能”“应用”；
            
    - 停用词过滤（移除 “的”“the”“and” 等无意义词汇）；
        
    - 词干提取 / 词形还原（如英文 “running”→“run”，用 Porter Stemmer/Lancaster Stemmer）；
        
    - 大小写统一、特殊符号去除。
        
- Step 2：构建倒排索引（Inverted Index） 全文检索的核心数据结构，将 “关键词” 映射到 “包含该关键词的文档 ID 及位置”，实现从 “词” 到 “文档” 的快速查询。 例如：关键词 “RAG” 对应文档 ID 1（位置 10-12）、文档 ID 3（位置 5-7），查询 “RAG” 时可直接定位到这两个文档，无需遍历所有数据。
    
- Step 3：关键词匹配与排序
    
    - 匹配阶段：根据用户查询的关键词，从倒排索引中快速筛选出包含这些关键词的文档（如查询 “Python 入门”，筛选出包含 “Python” 且包含 “入门” 的文档）；
        
    - 排序阶段：用 TF-IDF（词频 - 逆文档频率）、BM25（优化版 TF-IDF，更关注关键词在文档中的位置和密度）等算法计算 “查询 - 文档” 的相关性得分，返回得分 Top-K 的文档。
        

2. ### 常见开源方案
    

全文检索的开源生态成熟，工具多聚焦于 “高吞吐量” 和 “低延迟”，适合中小规模知识库（百万级文档以内）：

|   |   |   |
|---|---|---|
|开源工具|核心特点|适用场景|
|Elasticsearch|基于 Lucene 的分布式搜索引擎，支持水平扩展、实时索引，内置 BM25 排序算法|企业级知识库、日志检索、电商搜索|
|Apache Lucene|全文检索核心库（Elasticsearch 的底层依赖），轻量级、高性能，需自行开发分布式能力|定制化检索系统、嵌入式检索场景|
|Meilisearch|轻量级开源搜索引擎，部署简单（单二进制文件），支持模糊匹配、实时更新|小型项目、个人知识库、产品内搜索|
|Typesense|兼容 Elasticsearch API，性能更优（内存索引），支持地理空间检索|低延迟需求场景（如实时推荐、APP 搜索）|

3. ### 核心优势
    

- 精准定位关键词：能快速找到包含查询中特定关键词的文档，且可返回关键词在文档中的位置（如 “2023 年 GDP” 在文档第 5 段），便于后续 “上下文聚焦”；
    
- 效率极高：倒排索引的查询复杂度低（O (1) 级别的关键词定位），支持百万级、亿级文档的秒级召回；
    
- 可解释性强：召回结果的相关性可通过 “是否包含关键词”“关键词出现频率” 直接解释，便于问题排查（如未召回相关文档，可快速判断是否因关键词未匹配）；
    
- 对硬件要求低：无需高维向量存储，普通数据库（如 Elasticsearch、Solr，甚至 MySQL 的全文索引）即可支撑。
    

4. ### 局限性
    

- 无法理解语义，易受 “词不匹配” 影响：例如用户查询 “如何给宠物做清洁”，若文档中只有 “猫咪洗澡步骤”（无 “宠物”“清洁” 关键词），全文检索会漏召回；
    
- 对同义词、歧义句不友好：“手机” 和 “电话机” 是同义词，但全文检索会视为不同关键词；“苹果”（水果）和 “苹果”（公司）会被混淆召回；
    
- 对长查询处理效果差：若查询包含多个关键词（如 “2023 年中国一线城市 GDP 同比增速”），全文检索可能因部分关键词未匹配而漏召回（如文档只有 “2023 年北京 GDP 增速”，缺少 “一线城市”“同比”）。
    

## 二、向量检索（Vector Retrieval）

向量检索是基于 “语义相似性” 的现代检索方式，核心逻辑是将 “文本（或图像、音频）” 转化为高维向量（通过 AI 模型编码），通过计算 “查询向量” 与 “文档向量” 的相似度（如余弦相似度、欧氏距离），召回语义最接近的文档。其优势是能处理 “关键词不匹配但语义相似” 的场景（如 “治疗感冒的药” 与 “缓解流感症状的药物”）；劣势是可解释性弱（无法明确 “为何召回该文档”），且对向量模型和索引结构依赖高。

1. ### 技术实现：向量编码与近似最近邻（ANN）检索
    

向量检索的核心是 “将语义转化为可计算的向量”，并通过高效索引解决 “高维向量查询慢” 的问题，流程分为四步：

- Step 1：文本向量编码（Embedding） 用预训练的语言模型（如 BERT、Sentence-BERT、OpenAI Embeddings）将文档和查询转化为固定维度的向量（如 768 维、1536 维）。向量的每个维度代表文本的一个语义特征（如 “医疗”“感冒”“药物” 等隐性特征），语义越相似的文本，向量距离越近。 例如：“头痛怎么办” 和 “如何缓解头痛” 的向量余弦相似度接近 1，而与 “如何修电脑” 的相似度接近 0。
    
- Step 2：构建向量索引（ANN Index） 高维向量（如 1000 维）直接计算 “精确最近邻（Exact Nearest Neighbor, ENN）” 会消耗大量时间（O (n) 复杂度，n 为文档数），因此需构建近似最近邻（Approximate Nearest Neighbor, ANN）索引，在 “精度” 和 “速度” 间做平衡。常见 ANN 索引结构：
    
    - 聚类索引（如 FAISS IVF）：将向量聚类为多个 “桶”，查询时先定位到相似的桶，再在桶内计算精确相似度；
        
    - 图索引（如 HNSW）：将向量构建为多层图结构，查询时从顶层快速导航到相似向量，是当前主流（精度与速度兼顾）；
        
    - 哈希索引（如 LSH）：将高维向量映射到低维哈希值，通过哈希碰撞快速筛选候选向量，适合超大规模数据（十亿级）。
        
- Step 3：相似度计算与召回 接收查询后，先将查询编码为向量，再通过 ANN 索引快速筛选出 Top K 个相似向量对应的文档，最后返回给大模型。
    

2. ### 常见开源方案
    

向量检索的开源工具聚焦于 “高维向量效率” 和 “多场景适配”，覆盖从百万级到十亿级文档的需求：

|   |   |   |
|---|---|---|
|开源工具|核心特点|适用场景|
|FAISS (Facebook AI Similarity Search)|Facebook 开源，支持 CPU/GPU 加速，内置 IVF、HNSW 等多种索引，适合高维向量（100-10000 维）|科研场景、中小规模向量检索（千万级）|
|Milvus|分布式向量数据库，支持水平扩展、多索引类型（HNSW、IVF），兼容多模态数据（文本 + 图像）|企业级 RAG、多模态检索、推荐系统|
|Qdrant|轻量级向量数据库，支持地理空间检索、元数据过滤（如 “只召回 2024 年的文档”），部署简单|中小项目、需元数据筛选的 RAG 场景|
|Chroma|专为 RAG 设计的向量数据库，集成 Embedding 模型（如 Sentence-BERT），API 简洁，支持本地部署|开发者快速验证 RAG 原型、个人知识库|
|Weaviate|开源向量数据库，支持语义搜索、多模态，内置 GraphQL 查询接口，可关联结构化数据|需关联结构化数据的检索场景（如产品 + 文档检索）|

3. ### 核心优势
    

- 理解语义，解决 “词不匹配” 问题：例如用户查询 “如何给宠物做清洁”，向量检索能召回包含 “猫咪洗澡步骤”“狗狗梳理方法” 的文档（字面无 “宠物清洁”，但语义相关）；
    
- 支持多模态检索：除文本外，还可处理图片、音频、视频的向量（如用 CLIP 模型生成跨模态向量），适用于多模态知识库；
    
- 对文本语序、表达方式不敏感：“我想学习 Python” 和 “Python 学习教程在哪” 的向量相似度极高，召回结果一致。
    

4. ### 局限性
    

- 依赖高质量 Embedding 模型：模型效果直接决定向量质量，若模型对专业领域（如法律、医疗）的语义理解不足，会导致召回偏差；
    
- 缺乏 “细粒度匹配” 能力：无法精准定位查询中的关键词在文档中的具体位置（如用户问 “2023 年 GDP 数据”，向量检索可能召回包含 “GDP” 的文档，但无法快速确认是否有 “2023 年”）；
    
- 计算成本较高：高维向量的存储和相似度计算（尤其是未优化的暴力搜索）对硬件资源要求较高。
    

## 三、混合检索（Hybrid Retrieval）

混合检索是结合全文检索 “关键词精准性” 与向量检索 “语义关联性” 的融合方案，核心逻辑是：通过两种检索方式分别召回候选文档，再通过 “结果融合”（如加权投票、重排序）得到最终召回列表。其优势是兼顾 “精准匹配” 和 “语义泛化”，覆盖更多检索场景（如用户查询既含明确关键词，又有语义需求）；劣势是架构更复杂，需协调两种检索的结果冲突。

1. ### 技术实现：双检索并行与结果融合
    

混合检索的核心是 “如何高效并行两种检索” 和 “如何合理融合结果”，流程分为三步：

- Step 1：双检索并行召回 对用户查询同时执行 “全文检索” 和 “向量检索”：
    
    - 全文检索：按关键词匹配召回 Top K1 个文档（如 Top 50），保留排序分数（如 BM25 分数）；
        
    - 向量检索：按语义相似性召回 Top K2 个文档（如 Top 50），保留相似度分数（如余弦相似度）。
        
- Step 2：结果去重与对齐 去除两种检索结果中的重复文档（如同一文档既被关键词匹配，也被语义匹配召回），并将两种分数 “归一化”（如映射到 0-1 区间），避免因分数范围差异（如 BM25 分数可能是 0-10，余弦相似度是 0-1）导致权重失衡。
    
- Step 3：结果融合与重排序 两种主流融合策略：
    
    - 加权融合（Weighted Fusion）：给两种检索结果分配权重（如全文检索占 40%，向量检索占 60%），计算最终分数（最终分数 = 0.4×BM25 分数 + 0.6× 向量相似度），按分数排序；
        
    - 重排序模型（Reranking Model）：用小尺寸预训练模型（如 Cross-Encoder、BERT-small）对候选文档重新打分，模型输入为 “查询 + 文档”，输出语义匹配度，按该分数排序（精度更高，但耗时略增）。
        

2. ### 常见开源方案
    

混合检索的开源方案多基于 “全文检索工具 + 向量检索工具” 的组合，或提供一体化框架：

|   |   |   |
|---|---|---|
|开源方案|核心实现方式|适用场景|
|Elasticsearch + Milvus/Qdrant|独立部署全文检索（Elasticsearch）和向量检索（Milvus），通过代码实现结果融合|需灵活调整两种检索权重的场景|
|LangChain Retrieval|LangChain 框架提供 HybridSearch 类，支持 Elasticsearch（全文）+ FAISS（向量）的组合，内置加权融合逻辑|快速搭建 RAG 原型，需与 LangChain 生态联动|
|Haystack (Deepset)|开源 RAG 框架，支持 “Elasticsearch（全文）+ Milvus（向量）” 混合检索，内置 Cross-Encoder 重排序|企业级 RAG 开发，需重排序优化精度|
|LlamaIndex Hybrid Search|LlamaIndex 框架提供一体化混合检索，支持 FAISS（向量）+ SimpleVectorStore（全文），简化配置|基于 LlamaIndex 的 RAG 项目|

3. ### 核心优势
    

- 兼顾语义与字面相关性：既解决向量检索 “漏召回关键词匹配文档” 的问题，又解决全文检索 “漏召回语义相关文档” 的问题，召回精度显著高于单一策略；
    
- 鲁棒性强：对不同类型的查询（短查询、长查询、模糊查询、精确查询）均有较好表现，例如：
    
    - 短查询（“GDP 数据”）：全文检索可精准匹配 “GDP”，向量检索补充语义相关的 “经济指标” 文档；
        
    - 长查询（“如何用 Python 处理 Excel 数据”）：向量检索理解整体意图，全文检索定位 “Python”“Excel” 关键词；
        
- 灵活适配场景：可根据知识库类型（如纯文本、多模态）、查询需求（如精度优先、效率优先）调整融合权重和策略。
    

4. ### 局限性
    

- 系统复杂度高：需同时维护向量索引和全文索引，且融合策略需大量调参（如权重分配、Top-K 数量）；
    
- 性能开销增加：相比单一检索，需执行两次检索 + 一次融合，对系统响应速度有一定影响（需通过索引优化、缓存等方式缓解）。
    

## 四、三种检索方式的核心差异与选择建议

|        |                       |                               |                                    |
| ------ | --------------------- | ----------------------------- | ---------------------------------- |
| 对比维度   | 向量检索（Vector Search）   | 全文检索（Full-Text Search）        | 混合检索（Hybrid Search）                |
| 核心逻辑   | 语义相似性匹配               | 关键词字面匹配                       | 语义 + 关键词协同匹配                       |
| 相关性依据  | 向量相似度（余弦距离、欧氏距离）      | 关键词匹配度（TF-IDF、BM25）           | 向量相似度得分 + 关键词匹配得分                  |
| 检索效率   | 中（高维向量计算耗时）           | 高（倒排索引快速定位）                   | 中（两次检索 + 融合，需优化）                   |
| 适用查询类型 | 模糊查询、语义查询、多模态查询       | 精确查询、关键词查询、短查询                | 所有类型（短 / 长查询、精确 / 模糊）              |
| 适用知识库  | 多模态知识库、非结构化文本库        | 结构化文本库、关键词密集型库                | 混合类型知识库、通用知识库                      |
| 优势场景   | 需理解用户意图，如 “如何学 AI”    | 需精准定位关键词，如 “2023 GDP”         | 通用 RAG 场景，如客服、问答机器人                |
| 代表工具   | Milvus、Pinecone、FAISS | Elasticsearch、Solr、MySQL 全文索引 | Elasticsearch（多字段）+Milvus、Weaviate |
