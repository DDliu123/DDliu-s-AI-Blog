# Dify 平台架构深度拆解

> **项目地址：** <https://github.com/langgenius/dify>
>
> Dify 是一个开源的 LLM 应用开发平台，提供了一套直观的界面和强大的工具集，用于构建、部署和运营 AI 应用。它将 AI 工作流、RAG 管道、Agent 编排、模型管理及全面的 LLMOps 观测性功能整合在一个统一的平台上。

---

## 一、 Dify 整体架构概览

Dify 的架构设计层次分明，自上而下涵盖了从应用形态到技术底座的完整体系。

![[../Pictures/Pasted image 20260315165703.png]]

### 1.1 应用范式层：四种模式应对不同场景

Dify 提供了四种核心的应用开发范式，以满足不同复杂度的业务需求：

-   **Chatbot (对话助手)**：入门级解决方案，用于快速搭建自由对话式的智能问答系统。
-   **Agent (智能体)**：进阶型框架，赋予 AI 自主决策与执行能力。Agent 能基于用户意图进行动态推理，并自主调用工具完成复杂任务。
-   **Chatflow (对话流)**：支持上下文记忆的多轮交互系统，通过预设流程引导对话，适合客户服务、教育辅导等需要精确追踪对话状态的场景。
-   **Workflow (工作流)**：面向单次任务的自动化批处理引擎，适用于数据提取、内容生成等确定性、无需交互的场景。

这四种范式的核心差异在于**流程的控制权**：Chatbot 由用户主导，Workflow 由开发者预定义，Agent 由 AI 自主规划，而 Chatflow 则是介于两者之间的半结构化流程。

### 1.2 核心能力层：三大引擎驱动

-   **RAG 引擎**：知识库的核心。文档上传后，经过分块、清洗，支持向量搜索、全文搜索和混合搜索三种检索方式。同时，支持通过 Rerank 模型对检索结果进行二次排序优化，提升相关性。
-   **可视化工作流引擎**：通过拖拽节点编排复杂逻辑。节点类型丰富，包括知识检索、问题分类、条件分支、代码执行 (Python/NodeJS)、模板转换 (Jinja2) 等。其核心价值在于**将不确定性的 AI 输出，锁定在确定性的流程结构中**，从而显著提升应用的可控性和稳定性。
-   **Agent 框架**：Dify 的 Agent 支持两种推理模式，以适配不同的大语言模型：
    -   **Function Calling**：建议用于支持此模式的模型 (如 GPT-3.5/4)，性能更稳定。
    -   **ReAct (Reasoning and Acting)**：为不支持 Function Calling 的模型提供了一套可靠的替代框架。

### 1.3 扩展生态层：v1.0 的架构转折点

2025 年 2 月发布的 v1.0.0 版本是 Dify 的一个重要里程碑。它彻底重构了底层架构，将原本耦合在核心代码中的模型和工具全部插件化，形成了**“核心框架 + 插件生态”**的分层结构。

-   **插件化**：最具革命性的变化是引入了 Agent 策略插件，允许开发者自定义智能体的决策逻辑。配合新推出的 Marketplace，迅速汇聚了超过 120 个官方与社区插件。其热插拔系统采用 Python 的 `importlib` 动态加载机制，实现了插件的即插即用，更新插件无需重启服务。
-   **MCP 协议集成**：Dify 支持将构建好的工作流或 Agent 直接转换为标准的 MCP (Meta-Controller Protocol) Server，使其能被外部客户端访问。同时，Dify 也能通过 MCP 协议接入外部 API、数据库和服务，极大地简化了系统集成。

### 1.4 LLMOps 运营层：AI 应用的持续进化

Dify 的 LLMOps 模块使其区别于传统的应用开发平台。它提供了一套完整的工具链，用于监控、分析和迭代 AI 应用。

-   **核心理念**：AI 应用并非“开发完就结束”，而是需要基于生产数据进行持续迭代。
-   **运营闭环**：通过**标注回复**（人工编辑高质量答案以优化模型响应）、**Prompt IDE**（可视化调试和多模型对比测试）以及**内容审查**（敏感词过滤和安全策略），构成了一个完整的运营和优化闭环。

### 1.5 发布集成层：体现 BaaS (后端即服务) 策略

Dify 提供了灵活的部署和集成选项，开发者既可以直接使用其 Web 应用，也可以通过 API 将其能力集成到现有系统中，无需关心复杂的后端架构。

-   **API 优先**：所有功能都通过 API 暴露，方便二次开发和集成。
-   **Trigger (触发器)**：2025 年新增的事件驱动组件，允许 Dify 应用由定时任务、Webhook 事件或外部信号主动触发，使其不再局限于被动响应用户输入。

### 1.6 技术架构

Dify 采用模块化的微服务架构，各功能组件可独立部署和水平扩展。本地开发版通过 Docker Compose 一键启动，生产环境则推荐使用 Kubernetes 部署，并配合 PostgreSQL、Redis 和向量数据库以确保高性能和高可用性。

## 二、 核心模块深度拆解

### 2.1 模型管理

#### 核心设计：Provider + Model 两层插件体系

Dify 的模型管理架构设计得非常巧妙，其核心思想是**“资质”与“行为”分离**。

![[../Pictures/Pasted image 20260315165732.png]]

-   **供应商 (Provider)**：负责管理资质和凭据 (Credentials)。
-   **模型 (Model)**：负责具体的模型调用行为。

这种分离设计带来了极大的灵活性：同一个供应商可以挂载数十个不同能力的模型，而凭据只需在供应商层面配置一次。

#### 模型接入的三种方式

供应商支持三种模型配置方式，并且这三种方式可以共存：

1.  **预定义模型 (predefined-model)**：用户只需配置统一的供应商凭据，即可使用该供应商下的所有预定义模型。
2.  **自定义模型 (customizable-model)**：用户需要为每个模型单独配置凭据。例如，Xinference 同时支持 LLM 和 Embedding，但每个模型都有唯一的 `model_uid`，需要分别配置。
3.  **远程获取 (fetch-from-remote)**：只需配置统一的供应商凭据，模型列表通过该凭据从供应商远程拉取。例如，配置一个 OpenAI 的 `api_key` 后，Dify Runtime 就能获取到该账户下所有的微调模型。

这套架构的本质是一个**开放的模型接入标准**。任何新厂商只需编写一个 YAML 配置文件和一个 Python 类，就能以插件形式接入 Dify，无需改动核心代码。这正是 Dify 能够快速支持超过 200 个模型、并构建起繁荣模型生态的核心原因。

### 2.2 知识库管理

知识库是 RAG 应用的基石。Dify 在此模块的设计上同样体现了高度的灵活性和工程化的考量。

![[../Pictures/Pasted image 20260315164603.png]]

#### ETL 引擎：两套可切换的解析体系

这是知识库架构最关键的设计决策。Dify 提供了两套 ETL (Extract, Transform, Load) 引擎，可通过环境变量 `ETL_TYPE` 进行切换：

-   **内置 ETL (默认)**：使用 Python 原生库直接解析，轻量、无外部依赖，适合大多数常见文档。
-   **Unstructured ETL**：接入 `unstructured.io` 外部服务，擅长处理复杂版式（如图文混排、扫描件）和表格，但需要用户自行部署或调用其付费 API。

#### 文件格式解析逻辑详解

Dify 对不同文件格式采用了不同的解析策略。

![[../Pictures/Pasted image 20260315164649.png]]

-   **Word (.docx / .doc)**
    -   `.docx`：使用 `python-docx` 库，按顺序遍历段落和表格。表格会被转换为 Markdown 格式。
    -   `.doc`：旧格式无法直接处理，需调用系统命令 `antiword` 进行转换，这意味着运行环境的 Docker 镜像中必须预装该工具。
    -   **局限**：图片、批注、页眉页脚、艺术字等非文本内容会被完全忽略。

-   **Excel (.xlsx / .xls)**
    -   使用 `openpyxl` 库，按 工作簿 → 工作表 → 行 → 单元格 的层级遍历。每个工作表的数据被提取为一段文本。
    -   **局限**：公式只读取计算结果，不读取公式本身。合并单元格可能导致输出出现大量空列，建议上传前手动处理或转为 CSV。

-   **PPT (.pptx / .ppt)**
    -   使用 `python-pptx` 库，遍历每张幻灯片的所有形状 (Shape)，提取文本框中的内容。每张幻灯片的内容被视为一个独立段落。
    -   **优势**：演讲者备注 (Notes) 也会被提取，这部分内容往往很有价值。
    -   **局限**：图表、SmartArt、嵌入视频中的文字无法提取。`.ppt` 旧格式同样需要依赖外部工具 (如 LibreOffice) 进行转换。

-   **PDF (最复杂)**
    -   **内置 ETL 路径 (`pypdf`)**：能直接提取**文字型 PDF**的文本，但复杂排版（如多栏）可能导致文本顺序混乱。对于**扫描型 PDF**（本质是图片），提取结果为空，这是“上传了文件但知识库无内容”的常见原因。
    -   **Unstructured ETL 路径**：专为处理**扫描型 PDF**设计，能借助 OCR 提取文本，并更好地识别表格结构。
    -   **建议**：对于需要高质量提取表格的场景，建议先用 `tabula-py`、`camelot` 等专业工具在外部将 PDF 预处理成 CSV 或 Markdown，再上传到 Dify。

-   **Markdown & 纯文本**
    -   `.txt`, `.md`, `.html` 等纯文本格式直接以 UTF-8 编码读取。
    -   **优势**：Markdown 的标记（`#`, `**`, `|` 等）被完整保留。这反而成了优点，因为分块器可以利用这些标记（如 `
#`）作为天然的语义分隔符，实现更精准的切分。
    -   **业内共识**：**如果可以选择，优先上传 Markdown 格式**。将其他格式预处理成 Markdown 再导入，是提升 RAG 质量最简单有效的手段。

#### 知识库分块机制

分块是决定 RAG 效果的核心变量：切得太大，检索噪声多；切得太小，上下文信息断裂。Dify 在此提供了从算法到策略的完整解决方案。

![[../Pictures/Pasted image 20260315165520.png]]

1.  **底层算法：递归字符分割**
    Dify 的核心分块算法是**递归字符分割**，而非简单的按固定字数截断。它会按优先级（如 `

`, `
`, ` `）降序尝试使用语义分隔符进行切分。切分后，检查文本块长度（使用 GPT-2 的 Tokenizer 计算），若仍然过长，则继续用下一级分隔符递归处理。

2.  **通用分块模式：三种子策略**
    -   **自动分段**：系统自动推断最佳分隔符组合，无需用户配置，适合快速上手。
    -   **自定义分段**：用户可手动设置分段标识符（支持正则表达式）、最大长度和重叠长度，是生产环境中调优的主要手段。
    -   **Q&A 分段**：通过 LLM 为每个文本块生成问答对。检索时，匹配用户问题与预生成的问题，返回对应的答案段落。此策略能更好地弥合用户提问与原文之间的语义鸿沟，但预处理成本较高。

3.  **父子分块模式：解决精准与完整的矛盾**
    通用分块常常面临两难：要么块小而精准但上下文缺失，要么块大而完整但噪声过多。父子模式通过双层结构解决了这个问题：
    -   **子块 (Child)**：切分得很细（如一句话一个块），用于精准匹配用户查询。
    -   **父块 (Parent)**：包含完整的上下文段落。
    检索时，系统首先用查询匹配到最相关的**子块**，然后返回该子块对应的**父块**给 LLM，从而兼顾了检索的精准性和上下文的完整性。

4.  **关键参数与建议**
    -   **模式选择**：知识库的分段模式一旦选定便无法更改。建议在正式上传大量文档前，先用少量样本测试不同模式的效果。
    -   **性能考量**：父子模式的处理时间远高于通用模式。在算力有限的私有化部署场景中，需谨慎评估服务器配置。

### 2.3 Agent 智能体

Dify 的 Agent 框架是其最强大的功能之一，它赋予了 AI 应用自主思考和行动的能力。

![[../Pictures/Pasted image 20260315170829.png]]

#### Agent 执行流程与推理范式

1.  **入口与预检**
    当用户发起对话后，系统会执行一系列预检操作，其顺序经过精心设计：首先进行**敏感词检测**，避免不合规请求消耗后续资源；然后进行**标注回复匹配**，若命中则直接返回预设答案，节省 Token 成本；最后才加载工具、组织 Prompt，并调用模型。

2.  **策略选择：Function Calling vs. ReAct**
    这是 Dify Agent 最重要的工程设计之一。系统会根据当前选用的 LLM 是否支持 `TOOL_CALL`，**自动选择**推理策略：
    -   若支持，则使用 `FUNCTION_CALLING` 模式。
    -   若不支持，则自动切换到 `CHAIN_OF_THOUGHT` (CoT + ReAct) 模式。
    这个自动适配机制对开发者是透明的。当你在界面上将模型从 GPT-4 切换到其他不支持 Function Calling 的模型时，后台会无缝地从 `FunctionCallAgentRunner` 切换到 `CotChatAgentRunner`。

3.  **两种推理范式的内部逻辑**
    -   **Function Calling 模式**：核心是一个循环，LLM 在每一轮决定是否要调用一个或多个工具。如果需要，系统则执行工具调用，并将结果追加到上下文中，进入下一轮循环，直到任务完成或达到最大迭代次数。
    -   **ReAct 模式**：LLM 按照 `Thought -> Action -> Action Input -> Observation` 的格式进行“思考”。系统通过正则表达式解析出 `Action` (工具名) 和 `Action Input` (参数)，执行后将 `Observation` (结果) 返回给 LLM，供其进行下一步 `Thought`。

    ![[../Pictures/Pasted image 20260315170858.png]]

4.  **AgentThought：让推理过程可见**
    这是 Dify Agent 提升用户体验的关键设计。每一次迭代（思考、调用工具、获得结果），系统都会在数据库创建一条 `AgentThought` 记录，并通过 SSE (Server-Sent Events) 实时推送到前端。用户在界面上看到的“思考中… 调用工具…”的动态过程，就来源于此。这不仅增强了透明度，也为后续的调试和分析提供了宝贵数据。

5.  **安全机制：防止无限循环**
    为了防止 Agent 陷入工具调用的死循环而耗尽资源，系统硬编码了最大迭代次数（默认为 99 次）。在最后一次迭代时，系统会强制清空可用工具列表，迫使 LLM 必须生成最终答案，而不是继续尝试调用工具。

#### Agent 多轮对话与记忆机制

LLM 本身是无状态的，Dify 的记忆机制是在模型之上，通过工程化手段模拟出“有记忆”的体验。

![[../Pictures/Pasted image 20260315172042.png]]

1.  **根本原理：每次调用都“喂”入历史**
    Dify 的做法是：当用户发送新消息时，从数据库中取出该会话（由 `conversation_id` 标识）的**所有历史记录**，与当前消息一同拼装成完整的 Prompt，然后一次性发送给 LLM。LLM 看到了完整的对话历史，就仿佛“记得”之前的内容。

2.  **`conversation_id`：记忆的容器**
    每个会话都有一个唯一的 `conversation_id`。API 调用时是否传递此 ID，决定了是开启新对话还是在已有对话上继续。这解释了为什么“新建会话后 Agent 会忘记之前的约定”——这是设计使然，因为记忆是与 `conversation_id` 绑定的。

3.  **历史组织与 `TokenBufferMemory`**
    系统使用 `TokenBufferMemory` 来管理对话历史。为了控制成本和上下文长度，系统会对历史进行截断，遵循两个原则：
    -   **System Message (系统指令 + 工具列表) 永不截断**，因为这是 Agent 的“身份”和“能力”定义。
    -   **历史消息按从旧到新的顺序移除**，以保留最近的对话。

4.  **AgentThought 如何融入历史**
    这是 Agent 模式与普通对话模式在记忆机制上的最大区别。Agent 的历史记录不仅包含用户和助手的对话，还包含了**工具调用的完整思考链 (AgentThought)**。这意味着 LLM 在新一轮推理时，能看到上一轮“自己”调用了什么工具、得到了什么结果，从而实现基于前序工具结果的、更复杂的跨轮推理。

5.  **长期记忆：跨会话的两种实现**
    上述短期记忆会随着会话结束而清空。为了实现跨会-话的长期记忆，Dify 提供了两种机制：
    -   **知识库向量检索**：将被动地将用户信息、偏好等存入知识库。每次对话时，系统自动检索相关内容并注入到 System Prompt 中。
    -   **会话变量 (Chatflow 中)**：主动地在工作流中通过变量来读写和管理需要跨会话保留的特定信息。

#### 一个完整的三轮对话示例

```
# 第1轮：用户发起
User: "我叫张三，帮我查北京明天天气"
  → Dify 新建 conversation (id=abc)
  → Agent 组装 Prompt: [System] + [User: ...]
  → Agent 调用天气工具，写入 AgentThought
  → Agent 回答: "明天晴，25°C"，写入 Message 表

# 第2轮：用户追问 (传入 conversation_id=abc)
User: "那我需要带伞吗？"
  → Dify 读取 id=abc 的全部历史 (第1轮的 User+Assistant+AgentThought)
  → Agent 组装 Prompt: [System] + [历史] + [User: ...]
  → LLM 看到历史，知道天气是晴天，直接回答: "不需要带伞"

# 第3轮：用户继续 (传入 conversation_id=abc)
```
